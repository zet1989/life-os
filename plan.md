Техническое задание (ТЗ): Разработка экосистемы персональных AI-агентов (Life OS «Алексей & Co»)

1. Концепция и Цели проекта

Создание централизованной бэкенд-системы (Life OS), управляющей сетью специализированных Telegram-ботов. Система обладает единой реляционной базой данных и перекрестной долгосрочной памятью (RAG).
Цель: Автоматизация контроля здоровья, бизнес-аналитики (соло и партнерских проектов), психологического состояния, ремонта дома, обслуживания автомобиля, семейных процессов, а также трекинг жизненных целей и мечт с жестким разграничением прав доступа (ACL).

2. Стек технологий (Оптимизировано для Vibecoding)

Backend: Python 3.11+, фреймворк aiogram 3.x (поддержка мульти-ботов в одном event-loop).

AI Routing API: OpenRouter.

Тяжелые задачи и Vision: gpt-4o или claude-3.5-sonnet.

Рутина и парсинг текста: gpt-4o-mini или llama-3.

Распознавание голоса (Audio): OpenAI Whisper API (поддержка аудиосообщений для дневника, гаража и стройки).

Эмбеддинги (RAG): OpenAI text-embedding-3-small (размерность 1536, $0.02/1M токенов) — оптимальный баланс цена/качество для семантического поиска.

База данных (Реляционная): Supabase (PostgreSQL) — логи, транзакции, финансы, профили, КБЖУ, цели.

Векторная БД (RAG): Supabase pgvector — семантический поиск по дневникам, замерам, артикулам запчастей и бизнес-идеям.

Кэш и очереди: Redis — кэш контекста диалога (последние N сообщений, TTL 2 часа), дедупликация update_id от Telegram, очередь тяжёлых задач.

Хранилище файлов: Supabase Storage — хранение фотографий еды, чеков из строймагов и заказ-нарядов СТО.

Конфигурация: Pydantic Settings — загрузка и валидация секретов из .env (токены ботов, ключи API, Supabase credentials). Падает при старте если переменная не задана, а не при первом запросе юзера.

Отказоустойчивость: tenacity — exponential backoff retry для всех внешних API (OpenRouter, Whisper, Supabase). Без retry бот молча ляжет при 429/502/timeout.

Очередь тяжёлых задач: arq (async Redis queue) — Vision-анализ фото и Whisper-транскрипция занимают 3-10 сек. Бот мгновенно отвечает "⏳ Обрабатываю...", результат приходит отдельным сообщением.

Планировщик: APScheduler (AsyncIOScheduler) — cron-задачи: ежедневные сводки КБЖУ, напоминания о витаминах, еженедельные отчёты, ежемесячный аудит.

Логирование: structlog — структурированные логи для мониторинга и отладки.

Инфраструктура: Docker Compose (бот + Redis + worker), деплой на VPS (Beget), CI/CD через GitHub Actions + SSH.

3. Архитектура Данных и Безопасность (Фундамент)

3.1. Структура Базы Данных (Supabase Schema)

Для корректной работы мульти-проектности, финансов, настроек и целей создаются таблицы:

users (Пользователи):

user_id (PK, Telegram ID)

username (TEXT) — @username для удобства идентификации.

display_name (TEXT) — имя для отчётов и обращений бота.

role (admin, wife, partner)

permissions (JSONB: доступные боты, доступные проекты)

system_prompt_overrides (TEXT) — здесь хранятся динамические настройки (график витаминов, диета). Обновляется прямо через бота.

timezone (TEXT, default 'Europe/Moscow') — критично для cron-отчётов и корректных timestamp.

is_active (BOOLEAN, default TRUE) — мягкое удаление вместо DELETE.

created_at (TIMESTAMPTZ) — дата регистрации.

last_active_at (TIMESTAMPTZ) — обновляется при каждом сообщении для понимания активности.

projects (Бизнес и Личные проекты):

project_id (PK, SERIAL)

name (например, "KDK-GRUPP", "Cleaning Kostroma", "House Renovation", "Hyundai Sonata 2006", "Семейный бюджет")

type (solo, partnership, family, asset)

status (active, paused, archived) — для добавления/удаления проектов через бота.

owner_id (FK → users) — кто создал проект.

collaborators (BIGINT[]) — массив telegram_id партнёров с доступом к проекту. Партнёр видит проект, только если он в этом массиве. У owner — полный доступ всегда.

metadata (JSONB) — дополнительные данные (например, VIN авто, адрес дома, текущий пробег).

created_at (TIMESTAMPTZ).

goals (Цели и Мечты - "Северная Звезда"):

id, user_id

type (dream, yearly_goal, habit_target)

title (например, "Жить у океана 2 месяца в году", "Выйти из операционки клининга")

description (подробное описание видения)

status (active, achieved)

target_date (DATE) — дедлайн цели.

progress_pct (SMALLINT, 0-100) — процент выполнения.

parent_goal_id (FK → goals) — иерархия: мечта → годовая цель → привычка.

created_at, achieved_at (TIMESTAMPTZ).

events (Единая шина данных - Главная таблица):

id (BIGSERIAL), timestamp, user_id, project_id

bot_source (TEXT) — из какого бота пришло событие (health, garage, renovation, master, psychology).

event_type (meal, workout, thought, business_task, measurement, auto_maintenance, diary, habit)

raw_text (исходное сообщение или транскрипция аудио)

json_data (JSONB: структурированные данные: пробег, номера деталей, КБЖУ)

media_url

embedding (vector(1536)) — вектор pgvector для RAG-поиска прямо в таблице.

is_processed (BOOLEAN, default TRUE) — для отложенной обработки.

finances (Таблица для строгой математики):

id (BIGSERIAL), timestamp, project_id, user_id

transaction_type (income, expense)

amount (NUMERIC(12,2)) — строгие цифры для подсчета прибыли и стоимости владения авто/домом.

currency (TEXT, default 'RUB').

category (materials, marketing, salary, taxes, auto_parts, auto_service)

description

receipt_url (TEXT) — ссылка на фото чека в Supabase Storage.

source_event_id (FK → events) — связь с событием-источником.

conversations (Контекст диалогов с LLM):

id (BIGSERIAL), user_id, bot_source

role (system, user, assistant)

content (TEXT) — текст сообщения.

tokens_used (INT)

created_at (TIMESTAMPTZ)

Индекс: (user_id, bot_source, created_at DESC) — быстрая выборка последних N сообщений. При overflow (окно > 20 сообщений) — суммаризация через дешёвую модель.

api_costs (Контроль расходов на AI):

id, timestamp, user_id, bot_source

model (TEXT) — какая модель вызывалась.

tokens_in, tokens_out (INT)

cost_usd (NUMERIC(8,6))

task_type (TEXT) — meal_photo, transcription, embedding, diary...

Лимитер: жёсткий budget cap $ в день/месяц. При превышении — переключение на дешёвую модель.

reminders (Напоминания):

id, user_id, bot_source

message (TEXT)

cron_expression (TEXT) — например, '0 9 * * *' = каждый день в 9:00.

next_fire_at (TIMESTAMPTZ)

is_active (BOOLEAN)

Примеры: напоминание о витаминах, "Ты не записывал еду 5 часов", "Пора менять масло — прошло 8000 км", еженедельная сводка расходов.

model_routing (Маршрутизация AI-моделей):

task_type (PK, TEXT) — meal_photo, diary, financial_parse, transcription...

model (TEXT) — gpt-4o-mini, claude-3.5-sonnet...

max_tokens (INT), temperature (NUMERIC)

fallback_model (TEXT) — запасная модель.

Позволяет менять модели через БД без передеплоя.

3.2. Система разграничения прав (ACL) и Безопасность RAG

Изоляция входа и RAG: Партнёр (Александр) имеет доступ только к partnership-проектам, в которых он collaborator. RAG-запросы изолированы по project_id. Александр никогда не получит доступ к личным дневникам или расходам Алексея.

Роль wife: Доступ к ботам health и family (питание/спорт + семейный бюджет). Видит только свои данные (user_id-фильтрация) и семейные проекты (через collaborators). Нет доступа к бизнес-ботам, гаражу, дневнику, master-боту.

ACL middleware: Каждый запрос проходит через проверку: user_id → есть в таблице users? → is_active = TRUE? → есть ли права на этого бота? Если нет — молчать (не отвечать незнакомцам).

Идемпотентность: Telegram может прислать один update дважды. Хранить последние обработанные update_id в Redis, проверять перед обработкой. Без этого бот запишет трату дважды.

Контекст диалога: LLM не помнит предыдущие сообщения. Стратегия: последние 20 сообщений из таблицы conversations (кэширование в Redis, TTL 2 часа). При overflow — суммаризация старых сообщений через дешёвую модель.

Динамические промпты и сущности: Вы можете управлять проектами и настройками прямо в чате (например, команда /add_project или "Измени мою норму калорий").

4. Этапы разработки и Роли Агентов (Модульное внедрение)

ЭТАП 1: Здоровье, Спорт и Механика ввода (Агент «AI-Нутрициолог и Тренер»)

Задача: Обкатать Vision, аудио и динамическое изменение правил.

Пользователи: Алексей (admin) + Жена (wife). Оба имеют полный доступ к боту питания и спорта.

Разделение данных: Каждый пользователь видит только свои данные (events/КБЖУ фильтруются по user_id). У жены собственный system_prompt_overrides (своя норма калорий, диета, график витаминов). Сводки КБЖУ приходят каждому отдельно.

ACL жены: Роль wife имеет доступ ТОЛЬКО к боту health (питание/спорт). Нет доступа к бизнес-ботам, гаражу, дневнику, master-боту. permissions = {"bots": ["health"]}.

Функционал Питания: Парсинг фото тарелки -> расчет КБЖУ. Правила подтягиваются из БД (индивидуально для каждого user_id).

Функционал Спорта: Обработка текстовых/голосовых сообщений.

Изменение настроек: Кнопка "Настройки" -> Пишете команду боту -> Бот обновляет ячейку в БД (для конкретного user_id).

Ежедневная сводка: APScheduler отправляет вечернюю сводку КБЖУ каждому пользователю бота отдельно.

ЭТАП 2: Имущество и Активы (Дом и Авто)

Задача: Оцифровать обслуживание физических активов, избавиться от бумажных чеков и забытых артикулов.

2.1. Агент «AI-Прораб / Ремонт» (Проект "House Renovation")

Хранение замеров: Надиктовываете замеры. Бот сохраняет в pgvector.

Смета: Скидываете чек из строймага. Бот заносит сумму в finances (расход по дому).

2.2. Агент «AI-Механик / Гараж» (Проект "Hyundai Sonata 2006")

Контекст в промпте: Автомобиль Hyundai Sonata (сборка ТагАЗ, 2006 год). Специфика — нужны точные артикулы запчастей, отслеживание межсервисных интервалов.

Бортжурнал (RAG): Записываете голосовое из гаража: "Поменял масло в двигателе на пробеге 215 400 км. Залил ZIC 5W-40, фильтр MANN". Позже можно спросить: "Когда я последний раз менял масло и какое лил?", бот ответит мгновенно.

База знаний запчастей: Отправляете фото упаковки запчасти или пишете текст: "Сайлентблоки передних рычагов CTR номер CQKH-18, встали отлично". Эта информация улетает в векторную базу.

Учет расходов на авто: При отправке фото заказ-наряда из сервиса бот считывает итоговую сумму и работу, занося расход в таблицу finances (категория auto_service). Вы всегда будете знать реальную стоимость владения машиной за год.

ЭТАП 3: Мульти-проектный агент (Соло-бизнес и Маркетинг)

Задача: Гибкое управление любым количеством соло-проектов (КДК, Курсы и т.д.).

Формат: Личный бот Алексея.

Управление проектами:

Добавление/удаление проектов через чат (меняется статус в таблице projects).

Скидываете идею -> бот уточняет (Inline-кнопками): "К какому проекту это отнести: [КДК] [Курсы]?".

ЭТАП 4: Совместные боты

Задача: Три бота для совместной работы — бизнес-партнёрский учёт, стратегический коучинг и семейный бюджет. Все используют архитектуру collaborators[] для разграничения доступа.

4.1. Агент «Опердир» (бот partner) — Операционный учёт бизнес-партнёрства

Формат: Бот в общей группе (Ты + Александр + Бот) или в личных сообщениях.

Гибкое управление любым количеством совместных бизнес-проектов с партнёрами. Строгая аналитика прибыльности каждого проекта отдельно.

Мульти-проектная архитектура: Таблица projects имеет поле collaborators (массив telegram_id партнёров). Каждый участник видит только проекты, в которых он owner или collaborator. Это позволяет иметь одного бота на все партнёрские проекты (клининг, совместные бизнесы и т.д.), добавляя новые проекты через чат.

Управление проектами:

/add_project — создать новый партнёрский проект (только admin/owner).
/add_partner — добавить партнёра к проекту по Telegram ID.
/archive_project — архивировать проект через Inline-кнопки.
При записи расхода/дохода → бот предлагает Inline-кнопками: "К какому проекту: [Клининг] [Новый Бизнес]?".

Privacy mode: Включить в BotFather. Бот видит только команды и сообщения с @упоминанием или reply на его сообщение. Не реагирует на каждое сообщение в группе.

Логика группы: "Кто написал?" → определить user_id → проверить ACL → показать проекты, доступные этому юзеру.

Финансовая аналитика: Партнёр пишет о тратах → бот привязывает к проекту → SQL-запись в finances. Отчёт прибыльности по каждому проекту: доходы − расходы по категориям. Итоги считаются через математический запрос SUM, а не галлюцинациями LLM.

Изоляция данных: Партнёр (Александр) видит только partnership-проекты, в которых он collaborator. RAG-поиск изолирован по project_id. Никогда не получит доступ к личным/соло данным Алексея.

4.2. Агент «Бизнес-Ментор» (бот mentor) — Стратегический AI-коуч

Формат: Отдельный бот для личных сообщений каждому партнёру. Оба партнёра общаются с одним и тем же ботом, видят одни и те же partnership-проекты.

Персона: Бизнес-коуч мирового уровня, нацеленный на рост прибыли. Знает все ваши совместные проекты, помнит все обсуждения, идеи и решения. Даёт стратегические рекомендации, а не просто записывает — направляет к увеличению дохода.

Reply-клавиатура:
[💡 Идея] [🎙 Обсуждение]
[📊 Аналитика] [📁 Проекты]
[❓ Спросить]

Ключевые возможности:

Запись идей: Текст или голосовое → AI структурирует (суть, потенциал прибыли, первый шаг) → привязка к проекту через Inline-кнопки → сохранение в events + RAG embedding. При сохранении ментор сопоставляет с прошлыми идеями: "Эта идея перекликается с тем, что вы обсуждали 2 недели назад — см. запись..."

Анализ обсуждений (ключевая фича): Партнёры отправляют длинное аудио записи созвона/встречи → Whisper транскрибирует → LLM генерирует: (1) Краткое саммари обсуждения, (2) Список принятых решений, (3) Экшен-айтемы с ответственными. Всё сохраняется в events + RAG для последующих запросов. Можно потом спросить: "Что мы решили по ценам на прошлой встрече?"

Стратегическая память (RAG): Бот помнит ВСЕ идеи, обсуждения, решения. Поиск по базе знаний: "Какие идеи были по масштабированию клининга?", "Что Саша предлагал по маркетингу?". RAG фильтруется по project_id (партнёрские проекты).

Финансовая стратегия: Подтягивает реальные цифры из SQL (доходы, расходы, маржа), затем LLM анализирует тренды и даёт рекомендации по увеличению прибыли. Цифры — только из SQL, рекомендации — от LLM. Пример: "За последний месяц маржа клининга упала на 12%. Основная причина — рост расходов на расходники (+8000₽). Рекомендую: переговорить с поставщиком о скидке при объёме."

Отчёт за период: "Итоги за месяц по клинингу" → финансы (SQL) + ключевые решения (RAG) + прогресс по идеям + рекомендации.

Проактивные инсайты: При каждом ответе ментор видит контекст финансов и идей, может сам подсвечивать тренды и возможности без запроса.

4.3. Агент «Семейный бухгалтер» (бот family) — Семейный бюджет

Формат: Личный бот или бот в семейной группе. Алексей (admin) и жена (wife) имеют равный доступ.

Задача: Прозрачное управление семейным бюджетом. Оба супруга видят полную картину доходов и расходов, понимают куда уходят деньги и сколько остаётся.

Архитектура: Проекты с type = "family" и collaborators содержат telegram_id жены. RPC get_accessible_projects(user_id, "family") работает для обоих. Жена получает permissions = {"bots": ["health", "family"]}.

Reply-клавиатура:
[💰 Расход] [💵 Доход]
[📊 Отчёт] [📈 Категории]
[⚙️ Настройки]

Ключевые возможности:

Запись расходов/доходов: Текст или голосовое → LLM парсит сумму, категорию и описание → finances. Категории: продукты, ЖКХ, транспорт, здоровье, одежда, дети, развлечения, образование, подписки, прочее. Доходы: зарплата, бонусы, подработки, кэшбэк.

Фото чека: Фото из магазина → Vision OCR → авто-категоризация и сумма → finances.

Отчёт за период: "Итоги за месяц" → SQL SUM по категориям: общий доход − общий расход = баланс. Всё через SQL, никогда через LLM.

Топ категорий: "Куда ушли деньги в этом месяце?" → SQL GROUP BY category + сортировка по сумме.

Бюджетные лимиты: Установить лимит на категорию (например, продукты ≤ 40000₽/мес). Бот предупреждает при приближении к лимиту и при превышении.

Privacy mode (если в группе): Бот реагирует только на @упоминание или reply.

Изоляция данных: Жена видит только family-проекты. Бизнес-данные, активы, соло-проекты — недоступны.

Задача: Трекинг ментального состояния.

Агент «AI-Психолог»: Контроль отказа от вредных привычек.

Глобальный Дневник: Заполнение текстом и голосом. Все записи улетают в RAG для долгосрочной рефлексии.

ЭТАП 6: Master Intelligence (Главный Бизнес-Ментор и "Хранитель Видения")

Задача: Синтез данных и выравнивание жизни по Целям и Мечтам.

Формат: Твой главный личный бот (пульт управления).

6. Инфраструктура и Деплой

Хостинг: VPS (Beget) — нужен persistent process для aiogram.

Docker Compose: 3 контейнера — bot (все Telegram-боты), redis (кэш + очередь), worker (тяжёлые задачи Vision/Whisper).

Разработка: Long Polling (просто, не нужен домен). Продакшен: Webhooks + nginx + Let's Encrypt SSL.

CI/CD: GitHub Actions → SSH на VPS → `docker compose pull && docker compose up -d`. Автодеплой при push в main.

Секреты: `.env` файл (не в git). Минимум: BOT_TOKEN_HEALTH, BOT_TOKEN_ASSETS, BOT_TOKEN_BUSINESS, BOT_TOKEN_PARTNER, BOT_TOKEN_MENTOR, BOT_TOKEN_FAMILY, BOT_TOKEN_PSYCHOLOGY, BOT_TOKEN_MASTER, SUPABASE_URL, SUPABASE_KEY, OPENROUTER_API_KEY, OPENAI_API_KEY (Whisper + Embeddings), REDIS_URL.

Graceful shutdown: `on_shutdown` хук — при перезапуске сохранить состояние в Redis, не терять текущие обработки.

Health-check: Команда /status — проверка: БД жива? API отвечает? Redis доступен?

Экспорт данных: Команда /export — выгрузка всех данных юзера в JSON/CSV. Бэкап и портативность.

Аналитика в Telegram: Генерация PNG-графиков через matplotlib (тренды КБЖУ, расходы, прогресс целей) и отправка прямо в чат.
